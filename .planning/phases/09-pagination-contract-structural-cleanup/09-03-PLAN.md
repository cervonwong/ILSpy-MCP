## Why this plan is file-heavy (9 files) vs the 2-3 task guidance

The CONTEXT.md "small plan bias" guidance caps plans at ~6 files. This plan
touches 9 files (2 new, 2 renames = 4 source files, 2 DI sites, 1 new fixture,
2 test renames). That exceeds the soft cap by 3. Rationale for not splitting:

- **Atomic rename constraint.** C# type renames are a graph: tool → usecase →
  DI → tests. Splitting across plans leaves the tree in an un-buildable state
  between plans. Renames must land in one commit.
- **Test fixture is a dependency of the test renames.** Splitting the fixture
  creation into its own plan (e.g., "09-03a fixture, 09-03b rename") would
  still require 09-03b to wait on 09-03a and produces no parallelism win —
  just adds overhead.
- **Precedent velocity.** Phase 8 plan 08-01 landed DEBT-01 + DEBT-02 (a
  similar cross-layer normalize) in a single commit with 5 files; the Phase 9
  equivalent is roughly the same shape with 4 extra files from the one-time
  rename. Still well inside the Phase 8 velocity envelope (2-3 minutes, 3-6
  files average; this will be ~3-5 minutes, 9 files, ONE-TIME cost).
- **Task structure keeps concerns separate.** The plan is split into 3 tasks
  (fixture, rename+usecase+tool, tests) that execute in sequence inside one
  plan — the executor can reason about one thing at a time without the plan
  overhead of splitting them.

       int maxResults)

       // 1. Namespace exists but zero top-level types (rare — nested-only namespace)

       }

       {
           var rangeStart = offset + 1;
