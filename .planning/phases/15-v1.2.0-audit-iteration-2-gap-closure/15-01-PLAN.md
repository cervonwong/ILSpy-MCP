Rework `ScanILForStrings` (lines 176-222) from a single-pass inline scan to a two-phase scan so each ldstr hit can include a window of neighbouring instructions. Keep the existing signature, cancellation-safety, and `totalCount` semantics — only the internal shape changes.

**Window size:** N = 3 instructions before and N = 3 after each `ldstr` hit. Total window length up to 7 lines (3 before + the ldstr itself + 3 after), fewer when the hit is near the start or end of the method body. Document the choice in a `// Window size: N=3 before/after (OUTPUT-06)` comment above the constant.

**Phase 1 — record all instructions.** Before iterating matches, walk the BlobReader once and accumulate a `List<(int offset, string rendered)>` for every instruction. Render each instruction as `"IL_{offset:X4}: {opcodeName}"` with a minimal operand hint:
  - `ldstr "<value>"` — include the literal (use `reader.GetUserString` the same way the current scan does; truncate to 64 chars + `...` for safety against huge strings)
  - `ldc.i4 <n>`, `ldc.i4.s <n>`, `ldc.i8 <n>` — include the numeric value
  - `call`/`callvirt`/`newobj`/`ldfld`/`stfld`/`ldsfld`/`stsfld`/`ldflda`/`ldsflda`/`ldtoken`/`ldftn`/`ldvirtftn` — render as `"{opcode} token:0x{token:X8}"` (token is the 4-byte operand; cheap, no metadata resolution needed — keeps the scan fast and avoids coupling to Phase 12's resolver)
  - Branches (`br`, `br.s`, `brtrue`, etc.) — render as `"{opcode} IL_{target:X4}"` where target = currentOffset + instructionLength + branchOffset; if computing the target is awkward, fall back to `"{opcode}"` alone — the window is for orientation, not reassembly
  - All other opcodes — render as `"{opcodeName}"` (lowercase ECMA-style form, e.g. `nop`, `ret`, `ldarg.0`)
  - Use `opCode.ToString().ToLowerInvariant().Replace('_', '.')` for opcode name rendering (matches ECMA CIL spelling `ldarg.0` rather than `Ldarg_0`)

Advance the BlobReader using the existing `ILParsingHelper.SkipOperand` / `IsTokenReferenceOpCode` routines for correctness — do NOT re-derive operand sizes by hand. Stop on `ilReader.RemainingBytes == 0`.

**Phase 2 — detect matches and build windows.** Walk the accumulated instruction list; for each entry whose opcode is `ldstr` AND whose rendered value matches the regex, compute:
  - `start = max(0, i - 3)`
  - `end = min(list.Count - 1, i + 3)`
  - window slice = `list[start..=end]` rendered strings in order

Increment `totalCount` for every regex hit. Only add to the `results` collection while `results.Count < matchCap` (preserve the existing `offset + maxResults` cap semantics). Populate the new `StringSearchResult.SurroundingIL` field with the window slice.

Preserve every existing per-match field (`MatchedValue`, `DeclaringType`, `MethodName`, `MethodSignature`, `ILOffset`) — this is additive.

**Performance / DoS note:** the accumulation list is bounded by the method body instruction count (typically <1000 even for large methods). No extra allocation concerns vs the existing single-pass scan. Do NOT allocate per method body beyond one `List<(int,string)>`. Keep the outer cancellation checks intact.

Do NOT change `ScanILForConstants` — OUTPUT-07 already satisfied, scope is strictly OUTPUT-06.

Add a `private static string RenderInstruction(ILOpCode opCode, ref BlobReader reader, MetadataReader metadataReader)` helper (or equivalent) to keep `ScanILForStrings` readable. The helper reads the operand (advancing the reader) and returns the rendered line — Phase 1 calls it for every instruction.
