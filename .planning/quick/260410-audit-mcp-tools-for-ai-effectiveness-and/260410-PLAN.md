  <action>
Read every file in `Transport/Mcp/Tools/*.cs` (28 files — confirm exact count via `ls Transport/Mcp/Tools/*.cs`; CONTEXT.md mentions ~29, reconcile and report the actual number in the audit). For each tool, extract: tool name, `[Description]` attribute text, parameter list with descriptions, return type, and whether the tool has any pagination/limit parameters (e.g., `skip`, `take`, `maxResults`, `offset`).

Define the audit rubric (use these five themes — they map directly to CONTEXT.md > Specific Ideas):

1. **Naming** — Is the tool name self-explanatory? Does it follow a consistent verb-noun pattern (e.g., `find_*`, `get_*`, `decompile_*`, `search_*`, `list_*`, `analyze_*`)? Are there inconsistencies between similar tools?
2. **Scoping** — Is the tool's responsibility narrow and well-defined, or does it overlap with siblings? Are there pairs/triples that could merge or split?
3. **Descriptions (scenario-oriented)** — Does the `[Description]` express WHEN/WHY to use the tool (scenario language: "Use this when…", "For finding…") or only WHAT it does mechanically? Per CONTEXT.md, descriptions should express scenarios/use cases.
4. **Output richness** — Does the tool preemptively include nested references that an AI agent will obviously want next (e.g., does `disassemble_method` include the resolved metadata-token targets, type names, and member references inline)? Or does it force the AI into round-trips? Per CONTEXT.md: "Tools should include nested references by default (e.g., during disassembly) so AI agents have context for next steps." Balance: rich vs. context-flooding.
5. **Pagination (HARD RULE)** — Does any tool with potentially unbounded output expose pagination/limits? Per CONTEXT.md: "Any tool with unbounded output must have pagination — this is a hard rule." Flag every violation as a blocker for v1.1.

Score each tool per theme using a simple rubric (e.g., GOOD / WEAK / MISSING) — pick whichever 3-tier scheme reads cleanest. Document the chosen rubric in a "Methodology" section at the top.

Write the audit to `.planning/quick/260410-audit-mcp-tools-for-ai-effectiveness-and/260410-AUDIT.md` with this structure:

<narrative — group tools that share/break patterns; call out specific renames to consider>

<narrative — overlaps, gaps, candidates for merge/split>

<narrative — examples of good vs. mechanical descriptions; suggested rewrites for the worst offenders>

### 4. Output Richness
<narrative — which tools force round-trips; which already include nested context; specific examples>

### 5. Pagination (Hard Rule)
<narrative — list every tool with potentially unbounded output and whether it has pagination; explicitly call out violations as v1.1 blockers>
