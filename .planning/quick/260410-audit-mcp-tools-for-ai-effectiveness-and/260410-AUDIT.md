# MCP Tool Audit (260410)

## Methodology

- **Scope:** 28 MCP tools in `Transport/Mcp/Tools/*.cs`
  - **Reconciliation note:** CONTEXT.md says ~29 tools. Actual count from disk is **28** files. The discrepancy is a stale estimate; the audit covers everything currently registered.
- **Source per tool:** the `[McpServerTool(Name = ...)]` attribute, the `[Description]` attribute, the parameter list with their `[Description]` attributes, and the return type.
- **Themes (5):** Naming, Scoping, Description Quality, Output Richness, Pagination.
- **Rubric (3-tier):** `GOOD` / `WEAK` / `MISSING`.
  - `GOOD` = meets the principle without caveat.
  - `WEAK` = partially meets it; improvement is desirable but not blocking.
  - `MISSING` = fails the principle; improvement required.
  - `N/A` = the dimension does not apply (used only for Pagination on tools whose output is intrinsically bounded — e.g., a single type's hierarchy).
- **Priority labels (Summary Table):**
  - **P0** — Pagination hard-rule violation. Blocker for v1.1.
  - **P1** — Output forces obvious round-trips (lazy-agent failure).
  - **P2** — Description or naming polish.
  - **P3** — Already good; no action needed.
- **Out of scope:** This is a documentation pass. Zero source files in `Transport/Mcp/Tools/` are modified.

## Findings by Theme

### 1. Naming Consistency

The verb-noun convention is **almost** consistent, with a few stragglers worth flagging.

**Established verbs and their meaning (as inferred from current usage):**

| Verb              | Tools                                                                                                              | Implied meaning                                          |
| ----------------- | ------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------- |
| `analyze_*`       | `analyze_assembly`, `analyze_references`                                                                           | High-level / multi-step / opinionated analysis           |
| `decompile_*`     | `decompile_type`, `decompile_method`, `decompile_namespace`                                                        | Produce C# source / source-like output                   |
| `disassemble_*`   | `disassemble_type`, `disassemble_method`                                                                           | Produce CIL / IL listing                                 |
| `find_*`          | `find_usages`, `find_implementors`, `find_dependencies`, `find_instantiations`, `find_type_hierarchy`, `find_extension_methods`, `find_compiler_generated_types` | Locate things by structural relationship                 |
| `get_*`           | `get_type_members`, `get_type_attributes`, `get_member_attributes`, `get_assembly_attributes`, `get_assembly_metadata` | Read a single, well-defined slice of metadata            |
| `list_*`          | `list_assembly_types`, `list_embedded_resources`                                                                   | Enumerate everything of a kind                           |
| `search_*`        | `search_strings`, `search_constants`, `search_members_by_name`                                                     | Match against IL bytecode / metadata by pattern or value |
| `load_*`          | `load_assembly_directory`                                                                                          | Discover and ingest assemblies                           |
| `resolve_*`       | `resolve_type`                                                                                                     | Map a name to its defining assembly                      |
| `extract_*`       | `extract_resource`                                                                                                 | Pull binary/text content out                             |
| `export_*`        | `export_project`                                                                                                   | Bulk write to disk                                       |

**Inconsistencies / smells:**

1. **`decompile_namespace` is misnamed.** Its description literally says "Lists all types in a namespace... Returns a summary -- use decompile_type to get full source for individual types." That is `list_namespace_types` (or `list_types_in_namespace`), not a decompile operation. The current name will mislead an AI agent into expecting source code for an entire namespace at once.

2. **`find_compiler_generated_types` is the longest tool name in the project.** Acceptable because the noun is a real category, but it does break the "short verb + short noun" rhythm of its `find_*` siblings. Tolerable, no action.

3. **`analyze_references` overlaps with five `find_*` tools.** It is a dispatcher that routes to `find_usages`, `find_implementors`, `find_dependencies`, `find_instantiations` based on a string parameter. The verb `analyze_*` here means "I am a router," not "I produce opinionated analysis" — which conflicts with `analyze_assembly`'s usage of the same verb. See Scoping below.

4. **`decompile_type` vs `get_type_members`.** Names suggest different verbs but their descriptions overlap heavily ("Use this when you know the class name but need to see what methods and properties you can call on it" vs "Decompiles a class/interface/struct to understand what methods, properties, and constructors are available"). The split is real (one returns AI insights, the other returns a structured member list) but the names don't telegraph the difference. Consider `summarize_type` for the AI-insight variant.

5. **`load_assembly_directory` and `resolve_type` both take `directoryPath`.** They are a pair, but the noun differs (`load_*_directory` vs `resolve_*`). The pair would be more discoverable as `load_assembly_directory` + `find_type_in_directory` (or `resolve_type_in_directory`).

**Verdict:** Naming is broadly consistent. Two real fixes (`decompile_namespace` rename, `analyze_references` overlap) and three nice-to-haves.

### 2. Scoping & Responsibility

The biggest scoping question in the project is the relationship between `analyze_references` and the four `find_*` tools it dispatches to.

**The `analyze_references` problem.** Currently the project ships **both** the dispatcher (`analyze_references`) **and** the four individual tools (`find_usages`, `find_implementors`, `find_dependencies`, `find_instantiations`). This means an AI agent sees five tools doing essentially overlapping work. The dispatcher's description even tells the agent "use this for exploratory analysis when you're not sure which specific tool to call" — which is admitting the design is confusing. Pick one of the following:

- **Option A — Keep only the dispatcher.** Remove the four `find_*` tools. The agent has one entry point, picks `analysis_type`, done. Loses tool-name affordance (the agent has to know `find_usages` exists to look for it).
- **Option B — Keep only the four `find_*` tools.** Remove the dispatcher. Four sharp tools, each with a clear name. Agent discovers the right tool from the name. **Recommended.** This is more in line with the lazy-agent principle: clearer names mean fewer wrong-tool calls.

Either way the duplication is a v1.1 cleanup item.

**Other scoping observations:**

- **Attribute trio (`get_type_attributes`, `get_member_attributes`, `get_assembly_attributes`)** is well-scoped — each has a single, distinct level. Good.
- **Disassemble pair (`disassemble_type`, `disassemble_method`)** is well-scoped — type-level vs method-level IL. Good.
- **Decompile trio (`decompile_type`, `decompile_method`, `decompile_namespace`)** is well-scoped in principle but `decompile_namespace` is actually a list operation (see Naming).
- **`get_type_members` vs `decompile_type`** — overlap is real but defensible: one is structural-only (cheap, fast), the other does decompiler-driven AI summarization. Documenting the cost difference in both descriptions would help an agent pick correctly. Currently `get_type_members` does say "Faster than decompile_type when you just need to know 'what can I call?'" — `decompile_type`'s description doesn't have the inverse guidance.
- **`list_assembly_types` vs `decompile_namespace`** — both enumerate types. The former filters by namespace prefix and returns names, the latter returns names + signatures + member counts for one specific namespace. This is a real difference (cheap-list vs richer-list) but it's invisible from the names. Document it cross-link in both descriptions.
- **`find_extension_methods` vs `search_members_by_name`** — extension method discovery is genuinely a separate concern (it has to detect `[Extension]` attribute and the `this` parameter), so the split is justified. No action.
- **`load_assembly_directory` vs `resolve_type`** — these are a workflow pair: load first, resolve second. The descriptions don't cross-reference each other. Cross-linking would help the agent chain them.

### 3. Description Quality (Scenario-Oriented)

This is the most uneven theme in the audit. The project has **two distinct description styles** in active use:

**Style A — Scenario-oriented ("Use this when…")** — used by ~7 tools, mostly the older / heavily-curated ones:

- `analyze_assembly`: "Use this when you just installed a NuGet library and need to understand its overall architecture and entry points before writing code."
- `decompile_type`: "Use this when writing code that needs to interact with a NuGet library type but you don't know its structure."
- `decompile_method`: "Use this when you need to call a specific library method but don't understand its parameters, behavior, or side effects."
- `find_type_hierarchy`: "Use this when you need to understand inheritance relationships..."
- `find_extension_methods`: "Use this when you want to know what extension methods are available for a specific type."
- `get_type_members`: "Use this when you know the class name but need to see what methods and properties you can call on it."
- `list_assembly_types`: "Use this when you added a NuGet package but don't know what classes/types it provides."
- `search_members_by_name`: "Use this when you know what operation you want to perform (like 'Parse', 'Convert', 'Serialize') but don't know which type contains that method."

These are excellent. They tell an AI agent **when to reach for this tool** and what mental state the agent should be in to consider it.

**Style B — Mechanical ("Lists all X with their Y")** — used by ~21 tools:

- `disassemble_method`: "Get raw CIL/IL disassembly of a specific .NET method showing the complete IL instruction listing..."
- `disassemble_type`: "Get raw CIL/IL disassembly of a .NET type showing method signatures, fields, properties..."
- `find_usages`: "Find all call sites, field accesses, and property usages of a specific member across an assembly..."
- `find_implementors`: "Find all types implementing a given interface or extending a given base class..."
- `find_dependencies`: "Find all outward dependencies (method calls, field accesses, type references)..."
- `find_instantiations`: "Find all sites where a given type is instantiated (newobj) within an assembly..."
- `find_compiler_generated_types`: "Find compiler-generated types (async state machines, display classes, closures, iterators)..."
- `get_type_attributes`, `get_member_attributes`, `get_assembly_attributes`: "List all custom attributes..."
- `get_assembly_metadata`: "Get assembly metadata including target framework, runtime version, PE bitness..."
- `list_embedded_resources`: "List all embedded resources in an assembly with name, type, size, and visibility"
- `extract_resource`: "Extract embedded resource content. Returns text inline or binary as base64..."
- `export_project`: "Exports a .NET assembly as a complete C# project (.csproj + .cs files) to a target directory..."
- `load_assembly_directory`: "Load and list all .NET assemblies found in a directory..."
- `resolve_type`: "Resolve which assembly in a directory defines a given type..."
- `search_strings`: "Search for string literals in assembly IL bytecode matching a regex pattern..."
- `search_constants`: "Search for numeric integer constants in assembly IL bytecode..."
- `decompile_namespace`: "Lists all types in a namespace with full signatures..."
- `analyze_references`: "Unified cross-reference analysis tool. Routes to the appropriate analysis based on analysis_type parameter."

These tell the agent **what the tool does mechanically** but not **when to call it**. An AI agent reading the catalogue has to infer the use case from the mechanics, which is exactly the round-trip-and-fail-and-retry behavior CONTEXT.md flags as a problem.

**Suggested rewrites for the worst offenders:**

| Tool                          | Mechanical description (current)                                                | Scenario-oriented (proposed)                                                                                                                                |
| ----------------------------- | ------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `find_usages`                 | "Find all call sites, field accesses, and property usages of a specific member" | "Use this when you're about to change a method/field and need to know what will break. Returns every call site so you can assess impact before refactoring." |
| `disassemble_method`          | "Get raw CIL/IL disassembly of a specific .NET method..."                       | "Use this when the C# decompilation is misleading (compiler tricks, async state machines, generic specialization) and you need ground-truth IL behavior."   |
| `search_strings`              | "Search for string literals in assembly IL bytecode matching a regex pattern."  | "Use this to find hardcoded URLs, magic strings, error messages, or potential secrets in a compiled assembly. Useful for security auditing or migration."   |
| `find_compiler_generated_types` | "Find compiler-generated types..."                                              | "Use this when an assembly has mysterious `<>c__DisplayClass*` types and you need to map them back to the user code (lambdas, async methods, iterators) that produced them." |
| `search_constants`            | "Search for numeric integer constants in assembly IL bytecode."                 | "Use this to find magic numbers (status codes, buffer sizes, cryptographic seeds, registry constants) embedded in compiled IL when source is unavailable."  |
| `find_dependencies`           | "Find all outward dependencies..."                                              | "Use this when you need to know what an unfamiliar method touches before trusting it — what it calls, what fields it reads, what types it depends on."     |

**Rule of thumb:** every tool description should answer the question "**What is the user trying to do** when they need this?" — not "what does this tool produce?"

### 4. Output Richness

This theme is hard to assess from tool signatures alone (the actual output format lives in the use cases), but the tool surfaces still expose enough to flag concerns.

**Tools that almost certainly force round-trips (lazy-agent failure):**

1. **`disassemble_method` / `disassemble_type`** — IL disassembly references metadata tokens (e.g., `call instance string [System.Runtime]System.Object::ToString()`). The tool exposes a `showTokens` flag for raw token numbers but there is no flag to **resolve** the tokens to fully-qualified type names + assembly names + member kinds inline. An AI agent reading IL sees `call 0x0A000023` and has to make another tool call to figure out what that points to. This is the single most common round-trip waste in IL disassembly. Rich output here (resolve every metadata-token reference inline) is the highest-leverage v1.1 improvement.

2. **`find_usages` / `find_implementors` / `find_instantiations` / `find_dependencies`** — these return "matches" but the descriptions don't promise that each match includes (a) the containing type's full name, (b) the containing method's signature, (c) source line approximate location, (d) the surrounding IL context. If matches are bare names, the agent has to call `get_type_members` or `decompile_method` to figure out *which* `Process()` method this is. Rich output: each match should be self-describing.

3. **`find_dependencies`** — this is the worst offender of the four because dependencies are most useful when grouped (calls, field reads, type refs) and when each entry resolves to a fully-qualified name plus the assembly that defines it. If output is flat, the agent will pour the entire result back through `resolve_type` one item at a time.

4. **`search_strings` / `search_constants`** — match output should include the literal value, the containing method's full name, the IL offset (so the agent can correlate with `disassemble_method`), and ideally a small window of nearby IL (e.g., the next call instruction) so that `search_strings("Bearer ")` immediately tells the agent the surrounding code is calling `HttpClient.DefaultRequestHeaders.Authorization.Set(...)`. Without that context, every hit becomes a follow-up disassemble call.

5. **`get_type_members`** — returns method/property/field signatures. Should include the declaring type for inherited members, attribute summary for each member (e.g., `[Obsolete]`, `[Required]`), and a hint about whether the member calls anything dangerous (P/Invoke, file I/O). At minimum: which members are inherited vs declared, and which are virtual/abstract/sealed.

**Tools that already include nested context (good):**

1. **`find_compiler_generated_types`** — description explicitly says "with their parent method and type context." Good.
2. **`get_assembly_metadata`** — description says it includes "target framework, runtime version, PE bitness, strong name, entry point, **and all referenced assemblies**." Returning the reference graph in one call is exactly the right level of richness.
3. **`load_assembly_directory`** — returns "loaded assemblies with name and version, **and skipped files with reasons**." Good — telling the agent why something was skipped prevents a wasted retry.
4. **`get_type_attributes` / `get_member_attributes` / `get_assembly_attributes`** — each returns "constructor arguments and named properties." That's the right level — the agent doesn't have to make a follow-up call to figure out what the attribute was parameterized with.

**The flooding risk.** Rich output is good but bounded. Heuristics for v1.1:
- Default to a **summary view** with a `verbose=false` flag that opts into extra fields.
- For collection-returning tools, cap the inline detail to the first N items, then offer a continuation token.
- Resolve metadata-token references **lazily by default** (just include the resolved name) but offer a `resolveDeep=true` flag for full type signatures.

### 5. Pagination (Hard Rule)

This is the most actionable theme: **most tools that can return unbounded output have no pagination at all.**

**Tools with pagination (3 of 28):**

| Tool                | How                                                       |
| ------------------- | --------------------------------------------------------- |
| `search_strings`    | `maxResults` (default 100) + `offset` (default 0)         |
| `search_constants`  | `maxResults` (default 100) + `offset` (default 0)         |
| `extract_resource`  | `offset` + `limit` (for paginated binary extraction)      |

**Tools with a hard cap but no pagination (3 of 28):**

| Tool                  | Cap                                  | Problem                                                                                                                       |
| --------------------- | ------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------- |
| `decompile_namespace` | `maxTypes = 200`                     | If a namespace has 500 types, the agent gets the first 200 with no way to see the rest. Add `offset`.                          |
| `export_project`      | `maxTypes = 500`                     | This is not really a pagination problem (it's a one-shot export to disk), but the cap can silently truncate the export. The output should at least report "truncated: 500 of N types written." |
| `list_assembly_types` | (no explicit cap)                    | Returns potentially every type in an assembly. mscorlib has ~2500. No pagination, no cap.                                      |

**Tools that return potentially unbounded output and have NO cap and NO pagination — P0 violations:**

| Tool                            | Why unbounded                                                                              | Realistic worst case                                  |
| ------------------------------- | ------------------------------------------------------------------------------------------ | ----------------------------------------------------- |
| `list_assembly_types`           | One row per type in the assembly                                                           | mscorlib: 2500+ types                                 |
| `list_embedded_resources`       | One row per embedded resource                                                              | Most assemblies: <10. Some localization assemblies: 100s. |
| `find_usages`                   | One row per call site / field access / property reference of the member                    | A widely-used method like `ILogger.Log` in a large app: 1000s |
| `find_implementors`             | One row per implementing/inheriting type                                                   | `IDisposable` in a big app: 100s                      |
| `find_dependencies`             | One row per outward reference of the type/method                                           | A "kitchen-sink" service class: 100s                  |
| `find_instantiations`           | One row per `newobj` site                                                                  | A common DTO type: 100s                               |
| `find_extension_methods`        | One row per extension method targeting the type                                            | `IEnumerable<T>` in an app using LINQ + custom ext: 100s |
| `find_compiler_generated_types` | One row per compiler-generated type                                                        | Heavy LINQ + async assembly: 100s                     |
| `find_type_hierarchy`           | Bounded for a single type (its base chain + interfaces) — usually <20. **N/A.**            | -                                                     |
| `search_members_by_name`        | One row per matching member                                                                | searchTerm="Get" in mscorlib: 1000s                   |
| `disassemble_type`              | IL listing of every member of a type                                                       | A massive partial class: thousands of lines           |
| `disassemble_method`            | IL listing of one method — bounded by method length, but methods can be huge (machine-generated initializers, switch tables) | Codegen: 10k+ instructions                            |
| `decompile_type`                | C# source of one type — bounded by type size but unbounded in practice                     | Generated EF Core context: 1000s of lines             |
| `decompile_method`              | C# source of one method — same                                                             | Same as `disassemble_method`                          |
| `decompile_namespace`           | One entry per type in the namespace                                                        | `System.Collections.Generic`: 50+                     |
| `get_type_members`              | One row per member of the type                                                             | `string`: 100+ members                                |
| `analyze_references`            | Inherits unboundedness from whichever `find_*` it dispatches to                            | Same as above                                         |
| `analyze_assembly`              | Returns AI-generated narrative — bounded by LLM tokenization but realistically 1-5k tokens | Could explode for large assemblies                    |

**Tools where pagination is genuinely N/A:**

| Tool                       | Why                                                                |
| -------------------------- | ------------------------------------------------------------------ |
| `find_type_hierarchy`      | One type → its base chain + interfaces. Bounded by inheritance depth. |
| `get_type_attributes`      | One type → its attributes. Usually <10.                             |
| `get_member_attributes`    | One member → its attributes. Usually <5.                            |
| `get_assembly_attributes`  | One assembly → its attributes. Usually <30.                         |
| `get_assembly_metadata`    | One assembly → fixed-shape metadata record.                         |
| `extract_resource`         | Already has pagination via `offset`+`limit`.                        |
| `resolve_type`             | One type → matching assemblies. Usually 1, occasionally 2-3.        |
| `load_assembly_directory`  | Bounded by directory size. Could be unbounded for huge directories — borderline. Recommend adding pagination if directory scans return >100 assemblies. |
| `export_project`           | Writes to disk, not the wire. Cap is sufficient if it's reported.   |

**The hard rule, applied:** every P0 violation listed above must grow `(maxResults, offset)` parameters (or `(limit, cursor)`) before v1.1 ships. The output format must include a `truncated: bool` and `total: int` field so the agent knows whether to paginate.

- Resolve the `analyze_references` vs four `find_*` tools duplication. Recommendation: keep the `find_*` tools, drop the dispatcher.
